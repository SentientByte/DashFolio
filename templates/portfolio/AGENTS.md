# DashFolio agent instructions — `templates/portfolio/`

## Scope
These partials compose the main portfolio dashboard (KPI cards, holdings table, performance panels). They are included inside `portfolio_analysis.html` and expect well-structured context dictionaries generated by the calculation layer.

## Markup rules
- Keep table headers aligned with CSV/JSON export keys. If you rename a column in the template, mirror the change in any export or download helpers.
- Use `<th scope="col">`/`<th scope="row">` appropriately and ensure cells that display numeric data carry `text-end` to align figures.
- Wrap repeating card layouts in `{% for %}` loops and use `{% with %}` to shorten nested key access instead of writing long `foo.bar.baz` chains.
- Do not embed inline styles. New styling hooks should use BEM-style class names (`portfolio-status-pill`) and be defined in `static/css/app.css`.

## Data handling
- Guard against missing metrics: use `|default('—')` or conditional blocks so the dashboard remains stable when analytics are still loading.
- Currency values must be formatted through `format_currency` / `format_signed_currency` context helpers. Percentages should use `'%0.2f%%'|format(value)` or the dedicated formatting helper when available.
- When iterating through holdings, preserve the order provided by the context (often sorted by market value). Avoid resorting client-side unless there is a UX reason.

## Interaction hooks
- Any element that triggers a modal or AJAX update must include `data-*` attributes expected by the existing JavaScript utilities. If you add new hooks, document them in the surrounding template comments and ensure the TypeScript helpers know how to interpret them.

Adhering to these practices keeps the portfolio dashboard consistent with the backend data model and front-end utilities.
